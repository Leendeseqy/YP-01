## 1. Общее описание проекта

**Название системы:** EduGrader (Система оценивания учебных работ)  
**Тип проекта:** Учебный веб-проект  
**Область применения:** Автоматизация образовательного процесса в учебных заведениях  
**Срок реализации:** В рамках учебной практики

### 1.1. Краткое описание
EduGrader — это веб-система для автоматизации процесса проверки и оценивания учебных работ студентов. Система предназначена для цифровизации рутинных задач преподавателей и повышения прозрачности образовательного процесса.

**Ключевая ценность системы:**
- Сокращение времени на административные задачи преподавателей
- Стандартизация критериев оценивания
- Централизованное хранение учебных материалов и оценок
- Улучшение коммуникации между преподавателями и студентами

### 1.2. Бизнес-контекст
Проект разрабатывается в рамках учебной практики студента группы 21ИС-24. Система представляет собой прототип, демонстрирующий возможности автоматизации образовательных процессов.

## 2. Цели и задачи проекта

### 2.1. Стратегические цели
1. **Автоматизация:** Уменьшение ручного труда преподавателей на 40% при проверке работ
2. **Стандартизация:** Внедрение единых критериев оценки для всех студентов курса
3. **Прозрачность:** Обеспечение четкого понимания студентами системы оценивания
4. **Архивация:** Создание цифрового архива всех учебных материалов и оценок

### 2.2. Тактические задачи

#### Для учебного процесса:
- Создание цифровой среды для взаимодействия преподавателей и студентов
- Внедрение критериальной системы оценивания с использованием рубрик
- Организация удобного процесса сдачи и проверки работ
- Обеспечение своевременной обратной связи

#### Для разработки (учебные цели):
- Практическое применение знаний по веб-разработке
- Опыт работы с полным циклом разработки ПО
- Формирование портфолио учебного проекта
- Разработка документации по стандартам индустрии

## 3. Целевая аудитория и их потребности

### 3.1. Основные пользователи (Primary Actors)

**1. Преподаватель**
- **Демографические данные:** Возраст 30-65 лет, уровень цифровой грамотности средний/высокий
- **Основные потребности:**
  - Экономия времени при проверке работ
  - Единые стандарты оценки для всех студентов
  - Удобный интерфейс для предоставления обратной связи
  - Автоматическое ведение ведомостей
- **Критические требования:**
  - Простота освоения системы (минимальное обучение)
  - Надежность хранения данных
  - Возможность экспорта оценок

**2. Студент**
- **Демографические данные:** Возраст 17-25 лет, высокий уровень цифровой грамотности
- **Основные потребности:**
  - Понятная система критериев оценки
  - Своевременная обратная связь от преподавателя
  - Удобная загрузка работ в разных форматах
  - Доступ к истории своих работ и оценок
- **Критические требования:**
  - Мобильная адаптация интерфейса
  - Надежность загрузки файлов
  - Прозрачность сроков проверки

**3. Администратор системы**
- **Демографические данные:** Технический специалист, возраст 25-45 лет
- **Основные потребности:**
  - Централизованное управление пользователями
  - Мониторинг работы системы
  - Формирование отчетов для руководства
- **Критические требования:**
  - Инструменты для быстрого решения проблем
  - Система логирования событий
  - Резервное копирование данных

### 3.2. Вторичные пользователи (Secondary Actors)

**1. Руководитель учебного процесса**
- **Потребности:** Аналитика успеваемости, контроль качества преподавания
- **Требования:** Сводные отчеты, сравнительная аналитика

**2. Методист**
- **Потребности:** Анализ эффективности заданий, корректировка учебных программ
- **Требования:** Статистика по критериям оценки, выявление проблемных тем

## 4. Функциональные возможности системы

### 4.1. Модуль управления пользователями и доступом
**Цель:** Обеспечение безопасного и персонализированного доступа к системе

**Основные функции:**
1. Регистрация по приглашениям (инвайт-система)
2. Аутентификация через email/пароль
3. Восстановление доступа к аккаунту
4. Ролевая модель с тремя уровнями: администратор, преподаватель, студент
5. Управление профилями пользователей
6. Система сессий и безопасности

### 4.2. Модуль управления учебными курсами
**Цель:** Организация учебного процесса по курсам и группам

**Основные функции:**
1. Создание и каталогизация учебных курсов
2. Формирование учебных групп студентов
3. Назначение преподавателей на курсы
4. Управление академическими периодами (семестрами)
5. Архивация завершенных курсов

### 4.3. Модуль управления заданиями
**Цель:** Создание и настройка учебных заданий с четкими критериями

**Основные функции:**
1. Создание заданий различных типов:
   - Лабораторные работы
   - Проектные задания
   - Тестовые работы
   - Эссе и рефераты
2. Настройка системы рубрик (критериев оценки)
3. Установка дедлайнов и условий сдачи
4. Определение веса задания в итоговой оценке
5. Прикрепление учебных материалов и образцов

### 4.4. Модуль сдачи и проверки работ
**Цель:** Организация процесса загрузки, проверки и оценивания работ

**Основные функции:**
1. Загрузка студенческих работ:
   - Поддержка основных форматов (PDF, DOCX, ZIP)
   - Контроль размера файлов
   - Возможность повторной загрузки до дедлайна
2. Интерфейс проверки для преподавателей:
   - Система рубрик с весами
   - Возможность комментирования
   - Частичное оценивание
   - Вычисление итогового балла
3. Система уведомлений о статусе проверки

### 4.5. Модуль аналитики и отчетности
**Цель:** Предоставление данных для анализа учебного процесса

**Основные функции:**
1. Ведомости успеваемости по курсам и группам
2. Статистика по распределению оценок
3. Анализ выполнения критериев оценки
4. Экспорт данных в форматы Excel/PDF
5. Дашборды для преподавателей

### 4.6. Система коммуникации
**Цель:** Обеспечение своевременной коммуникации между участниками

**Основные функции:**
1. Автоматические уведомления:
   - О новых заданиях
   - О приближении дедлайнов
   - О выставленных оценках
   - О комментариях преподавателя
2. Внутрисистемные уведомления
3. Email-рассылка

## 5. Нефункциональные требования

### 5.1. Требования к производительности
- **Время отклика интерфейса:** Не более 2 секунд для 95% операций
- **Время загрузки файлов:** Зависит от размера, с индикацией прогресса
- **Количество одновременных пользователей:** Поддержка до 100 пользователей в рамках учебного проекта
- **Объем данных:** Хранение до 1000 учебных работ в рамках MVP

### 5.2. Требования к безопасности
- **Конфиденциальность данных:** Разделение доступа по ролям
- **Целостность данных:** Транзакционность критических операций
- **Аутентификация:** Надежная система проверки пользователей
- **Резервное копирование:** Ежедневные бекапы на этапе разработки

### 5.3. Требования к удобству использования
- **Интуитивность интерфейса:** Минимальное обучение для начала работы
- **Обучаемость:** Освоение основных функций за 15 минут
- **Мобильная адаптация:** Удобное использование на планшетах и смартфонах
- **Доступность:** Базовая поддержка стандартов доступности

### 5.4. Требования к надежности
- **Доступность:** 95% в рабочее время для учебного проекта
- **Восстановление:** Возможность восстановления данных из бекапа
- **Мониторинг:** Логирование ошибок и критических событий
- **Стабильность:** Отсутствие потери данных при стандартном использовании

## 7. Ограничения и границы проекта

### 7.1. Функциональные ограничения (версия 1.0)
1. **Первая версия (MVP) включает:**
   - Базовые модули: пользователи, курсы, задания, проверка

- **Поддержка:** В рамках учебной практики (3 месяца)

### 7.3. Ресурсные ограничения
- **Бюджет:** Учебный проект без коммерческого финансирования
- **Время:** Срок разработки ограничен периодом учебной практики
- **Команда:** Разработка одним студентом (автором документа)

## Приложение 1: Глоссарий терминов

**Рубрика (Rubric)** - Система критериев оценки, определяющая уровни выполнения задания  
**Критерий оценки** - Конкретный параметр, по которому оценивается работа  
**Дедлайн (Deadline)** - Крайний срок сдачи работы  
**MVP (Minimum Viable Product)** - Минимально жизнеспособный продукт
