# План тестирования системы портфолио и компетенций

## 1. Введение

### 1.1. Цель документа
Настоящий документ описывает план тестирования системы портфолио и компетенций. Цель тестирования - обеспечение соответствия системы функциональным и нефункциональным требованиям, выявление дефектов и подтверждение готовности к эксплуатации.

### 1.2. Область тестирования
Тестированию подлежат следующие компоненты системы:
- Графический пользовательский интерфейс (GUI)
- Бизнес-логика приложения
- Работа с базой данных PostgreSQL
- Генерация отчетов в формате DOCX
- Валидация входных данных

### 1.3. Объекты тестирования
- Основное приложение (`self_tracker.py`)
- База данных PostgreSQL с 8 таблицами
- Генерируемые отчеты в формате DOCX

## 2. Стратегия тестирования

### 2.1. Подходы к тестированию
1. **Функциональное тестирование**: проверка соответствия функциональным требованиям
2. **Интеграционное тестирование**: проверка взаимодействия компонентов
3. **Тестирование пользовательского интерфейса**: проверка GUI на удобство использования
4. **Тестирование производительности**: оценка времени отклика
5. **Тестирование надежности**: проверка устойчивости к ошибкам

### 2.2. Критерии начала тестирования
- Завершена разработка всех основных функций
- Готова тестовая база данных
- Установлены все необходимые зависимости
- Созданы тестовые данные

### 2.3. Критерии приостановки тестирования
- Обнаружены критические ошибки, блокирующие основные функции
- Недоступность тестовой среды
- Изменение требований в процессе тестирования

### 2.4. Критерии возобновления тестирования
- Устранены критические ошибки
- Восстановлена работоспособность тестовой среды
- Утверждены изменения требований

### 2.5. Критерии завершения тестирования
- Выполнены все тестовые сценарии
- Уровень успешного прохождения тестов ≥ 95%
- Отсутствуют открытые критические и блокирующие дефекты
- Уровень открытых дефектов низкой важности ≤ 5%

## 3. Виды тестирования

### 3.1. Функциональное тестирование
| Область тестирования | Методы тестирования | Инструменты |
|---------------------|---------------------|-------------|
| Добавление записей | Ручное тестирование, позитивные/негативные тесты | PyTest, ручные тесты |
| Работа с компетенциями | Автоматизированное тестирование | PyTest |
| Генерация отчетов | Ручное тестирование, проверка выходных данных | MS Word, ручные тесты |
| Система достижений | Автоматизированное тестирование | PyTest |
| Управление целями | Ручное и автоматизированное тестирование | PyTest, ручные тесты |

### 3.2. Интеграционное тестирование
| Компоненты | Методы тестирования |
|------------|---------------------|
| Приложение ↔ База данных | Тестирование транзакций, проверка целостности данных |
| GUI ↔ Бизнес-логика | Тестирование обработки событий |
| Модуль отчетов ↔ Данные | Проверка корректности экспорта |

### 3.3. Тестирование пользовательского интерфейса

## 5. План тестирования

### 5.1. Этап 1: Подготовка (2 дня)
**Задачи:**
1. Настройка тестовой среды
2. Создание тестовой базы данных
3. Подготовка тестовых данных
4. Установка инструментов тестирования

**Результаты:**
- Готовая тестовая среда
- Наполненная тестовая БД
- Установленные инструменты тестирования

### 5.2. Этап 2: Модульное тестирование (3 дня)
**Область:** Бизнес-логика приложения

**Тестовые случаи:**
1. Проверка валидации дат
2. Проверка расчета среднего уровня компетенций
3. Проверка логики присвоения достижений
4. Проверка обработки граничных значений

**Инструменты:** PyTest

### 5.3. Этап 3: Интеграционное тестирование (2 дня)
**Область:** Взаимодействие компонентов

**Тестовые случаи:**
1. Тестирование транзакций с БД
2. Проверка целостности связей между таблицами
3. Тестирование экспорта данных в Word
4. Проверка обновления статистики

**Инструменты:** PyTest, ручное тестирование

### 5.4. Этап 4: Функциональное тестирование (4 дня)
**Область:** Основные функции системы

**Тестовые случаи:**
1. Добавление всех типов записей
2. Работа с компетенциями и уровнями
3. Проверка системы достижений
4. Управление целями
5. Генерация отчетов

**Инструменты:** Ручное тестирование, чек-листы

### 5.5. Этап 5: Тестирование GUI (2 дня)
**Область:** Пользовательский интерфейс

**Тестовые случаи:**
1. Проверка макета и расположения элементов
2. Тестирование навигации между вкладками
3. Проверка валидации полей ввода
4. Тестирование сообщений об ошибках
5. Проверка доступности элементов

**Инструменты:** Ручное тестирование

### 5.6. Этап 6: Регрессионное тестирование (2 дня)
**Область:** Весь функционал

**Тестовые случаи:**
1. Повторение критических тестов
2. Проверка исправленных дефектов
3. Тестирование основных сценариев использования

**Инструменты:** Автоматизированные тесты, ручное тестирование

### 5.7. Этап 7: Приемочное тестирование (1 день)
**Область:** Соответствие требованиям

**Тестовые случаи:**
1. Проверка выполнения всех бизнес-требований
2. Тестирование основных пользовательских сценариев
3. Проверка документации

**Участники:** Заказчик, бизнес-аналитик

## 8. Управление дефектами

### 8.1. Классификация дефектов
| Уровень | Описание | Примеры |
|---------|----------|---------|
| Критический | Блокирует основные функции | Приложение не запускается, потеря данных |
| Высокий | Существенно влияет на функциональность | Не работает добавление записей, некорректный расчет |
| Средний | Влияет на удобство использования | Неверные сообщения об ошибках, проблемы с форматом |
| Низкий | Косметические проблемы | Опечатки, незначительные отклонения в UI |

### 8.2. Процесс обработки дефектов
1. **Обнаружение**: Тестировщик обнаруживает дефект
2. **Регистрация**: Дефект документируется в системе управления
3. **Анализ**: Определение уровня критичности
4. **Назначение**: Дефект назначается разработчику
5. **Исправление**: Разработчик исправляет дефект
6. **Проверка**: Тестировщик проверяет исправление
7. **Закрытие**: Дефект закрывается после успешной проверки

### 8.3. Шаблон отчета о дефекте
```
ID: [уникальный идентификатор]
Название: [краткое описание дефекта]
Описание: [подробное описание]
Шаги воспроизведения: [последовательность действий]
Ожидаемый результат: [что должно было произойти]
Фактический результат: [что произошло на самом деле]
Среда: [ОС, версия ПО, конфигурация]
Приоритет: [Критический/Высокий/Средний/Низкий]
Статус: [Новый/В работе/Исправлен/Закрыт]
```
