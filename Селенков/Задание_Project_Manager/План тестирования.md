# **План тестирования системы "Менеджер проектов"**

---

## **1. Введение**

### **1.1. Цель документа**
Данный документ описывает стратегию, объём, подход и расписание тестирования программного обеспечения "Менеджер проектов". План предназначен для команды тестирования, разработчиков и заинтересованных лиц.

### **1.2. Область тестирования**
Тестированию подлежат:
- Графический интерфейс пользователя (GUI)
- Функциональность управления проектами
- Работа с базой данных PostgreSQL
- Генерация отчётов (Excel, Word)
- Работа с файловой системой
- Обработка ошибок и исключений

### **1.3. Цели тестирования**
- Подтверждение соответствия функциональности бизнес-требованиям
- Проверка стабильности и отказоустойчивости системы
- Оценка удобства использования интерфейса
- Проверка корректности интеграционных компонентов

---

## **2. Стратегия тестирования**

### **2.1. Типы тестирования**
1. **Функциональное тестирование** — проверка бизнес-функций
2. **Интеграционное тестирование** — проверка взаимодействия компонентов
3. **Тестирование пользовательского интерфейса** — проверка GUI
4. **Тестирование производительности** — проверка времени отклика
5. **Тестирование отказоустойчивости** — обработка ошибок
6. **Регрессионное тестирование** — после изменений кода

### **2.2. Подходы к тестированию**
- **Ручное тестирование** — для GUI и пользовательских сценариев
- **Автоматизированное тестирование** — для модульных и интеграционных тестов (PyTest)
- **Black-box тестирование** — без знания внутренней реализации
- **White-box тестирование** — с учётом внутренней структуры кода

### **2.3. Критерии начала и окончания тестирования**
**Начало тестирования:**
- Завершена разработка всех основных функций
- Готовы тестовые среды и данные
- Утверждён план тестирования

**Окончание тестирования:**
- Выполнены все запланированные тесты
- Исправлены все критические и существенные дефекты
- Пройдены критерии приемки

## **7. Критерии качества**

### **7.1. Критерии приемки**
1. Все тесты TS-001 — TS-013 пройдены успешно
2. Покрытие кода тестами ≥ 80%
3. Отсутствие критических дефектов (Blocker, Critical)
4. Максимально 5 незначительных дефектов (Minor)

### **7.2. Метрики качества**
- **Покрытие кода:** ≥ 80%
- **Успешность тестов:** ≥ 95%
- **Время отклика UI:** < 2 секунд
- **Время генерации отчёта:** < 10 секунд

### **7.3. Дефекты**
**Уровни приоритета:**
1. **Blocker** — блокирует основную функциональность
2. **Critical** — критическая ошибка, но есть обходной путь
3. **Major** — существенная ошибка, влияющая на функциональность
4. **Minor** — незначительная ошибка, не влияющая на функциональность
5. **Trivial** — косметическая проблема
